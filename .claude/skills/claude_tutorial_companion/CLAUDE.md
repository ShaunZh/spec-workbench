# AI Agent 教程创作助手

这个项目为 Claude Code 提供一组面向 **AI agent 学习与开发过程教程化** 的本地规则与命令。

目标不是把零散开发记录简单改写成文章，而是把阶段性开发过程整理为：
- 教程大纲
- 单节教程正文
- 系列教程知识库

默认适用对象：
- 会 Python，但还没有系统做过 AI agent 开发的读者

默认输入素材：
- 开发笔记
- 代码片段
- 当前会话中关于实现过程的说明
- 本地 Markdown 草稿

默认工作方式：
- 阶段汇总型
- 不要求每一步都记录，但在一个阶段结束后集中整理

## 总体原则

1. 先还原开发事实，再进行教学表达。
2. 任何教程内容都必须回答四件事：
   - 做了什么
   - 为什么这么做
   - 遇到了什么问题
   - 最后得到什么经验
3. 优先保留设计动机、关键取舍、失败尝试、调试路径。
4. 不要把半成品包装成“最佳实践”。要明确哪些部分已经验证，哪些只是当前可行方案。
5. 输出内容应面向教学，而不是仅面向自我备忘。
6. 如果输入不足以支撑教程化表达，明确列出缺失信息，不要擅自编造。

## 教程化提炼规则

当你收到开发笔记、代码、草稿或对当前阶段的说明时，按以下顺序处理：

### 第一步：识别阶段主题
先判断这一阶段更接近哪一类：
- 概念理解：如 tool calling、memory、planning、agent loop
- 最小实现：如第一个可运行 demo
- 系统设计：如状态管理、工具抽象、模块边界
- 调试修复：如 bug 排查、失败原因、修复方案
- 工程化整理：如重构、测试、评估、部署

### 第二步：抽取事实材料
从输入中整理出：
- 目标
- 当前实现
- 关键代码
- 关键报错或障碍
- 排查动作
- 最终结果
- 尚未解决的问题

### 第三步：转成教学表达
将事实材料映射为：
- 本节要解决的问题
- 读者需要的前置知识
- 关键概念解释
- 步骤化实现过程
- 常见误区 / 踩坑点
- 一句话经验总结

### 第四步：判断输出类型
根据用户请求选择输出：
- 教程大纲
- 单节教程
- 知识库条目
- 衔接建议（下一节写什么）

## 输出质量标准

### 教程大纲
教程大纲应包含：
- 系列标题
- 目标读者
- 学习目标
- 章节列表
- 每章一句话定位
- 推荐顺序
- 当前已覆盖 / 待补充内容

### 单节教程正文
单节教程默认结构：
1. 本节解决什么问题
2. 为什么需要这一步
3. 实现思路
4. 关键代码与解释
5. 调试与踩坑
6. 最终结果
7. 本节小结
8. 下一步建议

### 知识库条目
每条条目默认结构：
- 标题
- 类型（概念 / 模式 / 坑点 / 决策 / 实践）
- 背景
- 核心结论
- 证据或代码依据
- 适用边界
- 可链接章节

## 写作风格要求

1. 默认用中文输出。
2. 面向“懂 Python、但没系统做过 agent 的开发者”。
3. 解释原因优先于堆砌术语。
4. 代码说明要围绕“这段代码解决了什么问题”。
5. 保留必要技术细节，但避免空泛鸡汤式总结。
6. 能明确不确定性时，就明确不确定性。

## 缺失信息处理

如果素材不足，优先输出：
- 当前能确定的教程框架
- 明显缺失的关键信息
- 建议补充的最小材料

不要为了凑完整教程而捏造：
- 未提供的实现细节
- 未发生过的调试过程
- 未验证的效果结论

## 推荐命令

本目录下提供三个命令：
- `/tutorial-outline`：从阶段材料生成教程大纲
- `/tutorial-section`：从某一阶段生成单节教程正文
- `/tutorial-kb-update`：把阶段结论沉淀为系列知识库条目

## 建议用法

### 场景 1：一轮开发后做阶段总结
把本轮的开发笔记、代码片段、问题与结论贴给 Claude，然后运行适合的命令。

### 场景 2：已有零散 Markdown 草稿
把草稿内容作为输入，让 Claude 统一整理结构、补足教学表达，并指出缺失。

### 场景 3：维护系列教程
每完成一节或一个主题，运行知识库更新命令，把经验沉淀成可复用条目，再反向更新系列大纲。
