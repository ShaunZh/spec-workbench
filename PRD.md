# PRD：AI 需求分析工作台
版本：v0.2  
阶段：MVP  
文档状态：可用于 Day 1 开发  
项目类型：产品型项目 + 学习型项目  
开发方式：开发者主导，AI 协作辅助

---

# 1. 项目背景

在日常研发协作中，需求描述、Bug 说明、客户反馈、PRD 片段往往存在以下问题：

- 信息不完整
- 缺少结构化整理
- 风险点和边界条件不明确
- 验收标准模糊
- 团队沟通成本高

传统聊天式 AI 工具虽然能回答问题，但往往存在这些不足：

- 输出不稳定
- 缺乏结构化结果
- 无法沉淀为项目资产
- 工具调用过程不可见
- 缺乏适合研发场景的 UI

因此，本项目希望构建一个面向研发协作场景的 **AI 需求分析工作台**，帮助用户将原始文本快速转化为结构化分析结果，并保留对话与分析历史。

与此同时，本项目不仅是一个产品原型，也是一项 **面向开发者自身成长的学习型项目**。  
项目发起人的核心目标之一，是通过亲自设计、实现和迭代该项目，系统学习 AI agent 开发相关能力，包括但不限于：

- LLM 应用开发
- structured output 设计
- tool calling
- agent workflow 编排
- prompt 设计与迭代
- AI 协作开发流程
- AI 应用的评测与调试

因此，本项目在产品价值之外，还承担“学习载体”和“实践样本”的作用。  
在后续开发过程中，项目将采用 **开发者 + AI 协作开发** 的方式推进，即利用 Claude / ChatGPT / Cursor 等 AI 工具协助完成脚手架、样板代码、文档和部分实现，同时由开发者主导核心设计、架构取舍、prompt/tool/workflow 定义和最终质量控制。

# 2. 产品目标

## 2.1 核心目标
提供一个 Web 端 AI 工作台，支持用户输入需求文本、Bug 描述或反馈内容，并生成结构化分析结果，包括：

- 摘要
- 待办项
- 风险点
- 验收标准
- 待确认问题

## 2.2 MVP 目标
在第一阶段，仅验证以下能力：

1. 支持聊天式输入与流式回复
2. 支持“分析模式”输出固定结构结果
3. 支持至少一个工具调用
4. 支持会话历史保存与查看
5. 支持分析结果导出为 Markdown

## 2.3 非目标
MVP 阶段暂不包含：

- 登录鉴权
- 多用户协作
- 文件上传解析
- 向量检索 / RAG
- 多 Agent 编排
- 复杂权限系统
- 多租户
- SaaS 计费
- 复杂任务规划
- 长期记忆
- 生产级监控平台

# 3. 学习目标

本项目除了产品交付目标外，还承担以下学习目标。

## 3.1 学习目标 1：掌握 AI 应用开发基础
通过项目实践掌握以下能力：

- LLM API 接入
- 流式输出处理
- 前后端 AI 应用集成
- 基础会话管理

## 3.2 学习目标 2：掌握 Agent 核心机制
通过逐步实现分析模式和工具调用，掌握：

- structured output
- tool calling
- 工具结果与模型结果融合
- 多步任务处理基础
- 单 Agent 场景下的基础 workflow

## 3.3 学习目标 3：建立 AI 协作开发方法
在项目开发过程中，形成一套可复用的 AI 协作开发方式，包括：

- 如何把 PRD、数据结构、接口定义转化为高质量提示词
- 如何让 AI 生成脚手架与样板代码
- 如何 review AI 生成代码
- 如何把核心设计权保留给开发者
- 如何利用 AI 提高开发效率但避免过度依赖

## 3.4 学习目标 4：沉淀可复用经验
项目完成后，希望输出以下可复用成果：

- 一套适合 AI 项目开发的提示词模板
- 一套 AI 协作开发流程
- 一套 AI agent 项目从 MVP 到迭代的实践经验
- 一个可写入简历和作品集的项目案例

## 3.5 学习边界
虽然本项目承担 AI agent 学习目标，但在 MVP 阶段不追求覆盖所有 agent 技术主题。  
第一阶段学习重点仅包括：

- LLM API 接入
- structured output
- 单工具调用
- 基础 workflow
- AI 协作开发流程

MVP 阶段暂不作为重点学习内容的包括：

- 多 Agent 编排
- 长期记忆
- 复杂任务规划
- 自主任务执行系统
- 大规模评测平台
- 复杂权限与生产级多租户架构

这样做的目的是保证项目节奏清晰，优先通过一个小而完整的项目建立 AI agent 开发的基础认知。

# 6. 主要功能

## 6.1 聊天模式
用户输入任意文本，AI 以普通对话方式进行回答。

### 输入
- 用户文本

### 输出
- 流式自然语言回复

### 价值
- 提供自由问答能力
- 用于补充说明、追问、扩展分析

---

## 6.2 分析模式
用户输入一段需求文本、Bug 描述或反馈内容，系统生成结构化分析结果。

### 输出字段
- `summary`
- `todos`
- `risks`
- `acceptance_criteria`
- `open_questions`

### 价值
- 形成统一结构输出
- 便于后续开发和测试使用

---

## 6.3 工具调用：spec_score
系统提供一个工具，用于检查输入需求的完整性。

### 功能
根据规则检查文本中是否包含：

- 用户角色
- 动作行为
- 业务目标
- 验收标准
- 边界 / 异常情况

### 输出
- 完整性评分
- 缺失项列表
- 改进建议

### 价值
- 让分析过程更可信
- 体现 AI 与规则工具结合的工作流能力
- 作为学习 tool calling 的最小可用样本

---

## 6.4 会话历史
系统支持保存和查看历史会话。

### 功能
- 创建会话
- 查看会话列表
- 查看单个会话详情
- 查看历史分析结果

### 价值
- 支持追溯
- 避免分析结果丢失

---

## 6.5 Markdown 导出
支持将本次分析结果导出为 Markdown 文本。

### 价值
- 可复制到 PRD、飞书文档、Notion、GitHub issue 等场景

---

# 7. MVP 用户流程

## 7.1 聊天模式流程
1. 用户打开工作台
2. 输入一段问题或描述
3. 系统流式返回回答
4. 会话自动保存

## 7.2 分析模式流程
1. 用户切换到“分析模式”
2. 粘贴需求文本
3. 系统调用 AI 分析
4. 必要时调用 `spec_score` 工具
5. 页面右侧展示结构化分析结果
6. 用户可导出 Markdown

# 11. 关键交互要求

## 11.1 流式输出
AI 回复需逐步展示，避免长时间空白等待。

## 11.2 工具调用可见
用户应能看到：

- 工具开始
- 工具结果
- 最终回复

## 11.3 分析结果独立展示
结构化分析结果不能只埋在聊天文本里，必须在右侧面板单独展示。

## 11.4 错误可恢复
出现错误时，至少支持：

- 错误提示
- 重试

## 11.5 学习过程可追踪
为了服务开发者学习目标，关键设计与实现过程应尽量文档化，包括：

- schema 定义
- prompt 设计
- tool 输入输出
- 重要架构取舍
- AI 协作开发提示词

# 13. 风险与约束

## 13.1 风险
- 模型结构化输出不稳定
- SSE 流式实现容易出错
- 工具调用链展示不够清晰
- 结果质量高度依赖 prompt 设计
- AI 生成代码可能带来过度设计或隐藏问题
- 学习目标和产品目标并行时，容易导致范围膨胀

## 13.2 约束
- 第一阶段优先追求可用，不追求架构完美
- 使用 SQLite，降低开发复杂度
- 不引入过多框架和抽象层
- 所有 AI 生成代码必须经过开发者 review
- 先完成最小闭环，再进入下一阶段迭代

# 14. 开发方式说明

本项目采用 **开发者主导、AI 协作辅助** 的开发方式推进。

## 14.1 原则
- AI 负责提升开发效率，优先承担脚手架、样板代码、基础 CRUD、文档初稿、测试样例等重复性工作
- 开发者负责核心设计，包括产品边界、数据结构、prompt、tool、workflow、交互体验、错误处理和最终质量控制
- 所有 AI 生成代码都必须经过开发者 review 和必要修改后才能纳入项目

## 14.2 协作目标
通过该项目，不仅交付一个可演示的 AI 工作台，还要验证以下问题：

- 如何高效地用 AI 搭建项目骨架
- 如何把 PRD 转化为高质量实现提示词
- 如何在 AI 协作开发中保持架构一致性
- 如何避免 AI 生成代码导致的过度设计和质量失控

## 14.3 预期产出
- 项目代码
- 提示词模板
- AI 协作开发记录
- 项目复盘文档

## 14.4 分工原则

### 适合交给 AI 的内容
- 项目初始化
- 目录结构搭建
- 样板代码
- 基础 CRUD
- README 初稿
- mock 数据
- 测试样例初稿

### 适合开发者主导的内容
- 产品范围与节奏
- 数据结构设计
- structured output schema
- prompt 设计
- tool 定义
- workflow 设计
- UI/UX 细节
- 错误处理与边界情况
- 最终代码 review 与质量把关

# 15. MVP 验收标准

满足以下条件即视为 Day 1 之后可继续推进：

- 前后端项目骨架创建完成
- 首页三栏布局可运行
- FastAPI 服务可启动
- `GET /healthz` 可访问
- 已具备 conversations 基础接口骨架
- 前后端目录结构清晰
- 环境变量模板已准备

MVP 全阶段完成时，应满足：

- 能完成一次完整的需求分析流程
- 能展示结构化分析结果
- 能展示一次工具调用过程
- 能保存和查看历史会话
- 能导出 Markdown
- 能作为 AI agent 学习案例进行讲解和复盘
